# 業種別横展開 md 作成ガイド

## このガイドの役割
このガイドは、各業種の計画 md を同じ品質で作るための共通ルールです。  
対象は次の3者です。

- 業種担当者: 現場の業務妥当性を判断する
- Web担当者: 提案資料・営業導線・訴求整理を判断する
- UI/UX デザイナー: 情報設計・見せ方・端末別体験を判断する

参照テンプレート: [`industry-rollout-template.md`](./industry-rollout-template.md)

## 1. 書き始める前に確認すること
- [`lib/industry-profiles/index.ts`](../lib/industry-profiles/index.ts) の `dashboardTitle`、`labels.*`、`kpiLabels`、`statusLabels` を確認する
- [`lib/industry-page-hints.ts`](../lib/industry-page-hints.ts) の `IndustryPageHints` で差し替え対象ページを確認する
- [`lib/app-template-config.ts`](../lib/app-template-config.ts) の `topNav`、`bottomNav`、`extensionSlots` を確認する
- [`components/navigation/mobile-parent-back-link.tsx`](../components/navigation/mobile-parent-back-link.tsx) と [`components/page-tag-links.tsx`](../components/page-tag-links.tsx)、[`components/app-shell.tsx`](../components/app-shell.tsx) を見て、モバイルの一覧／詳細の導線を把握する
- [`lib/feature-demos-catalog.ts`](../lib/feature-demos-catalog.ts) を見て、その業種で使う技術デモの意味づけを決める
- 先に [`industries/staffing-reference.md`](./industries/staffing-reference.md) を読み、何を守り、何を変えるかを明確にする

## 1-1. 最重要ガードレール
- `staffing` は基準サンプルであり、他業種対応の過程で原則変更しない
- 共通ファイルを触る場合は「対象業種だけに効く分岐」または「既存 `staffing` と完全互換の既定値」を前提にする
- md の段階で、必ず「`staffing` に影響しない理由」を1段落で書く
- 実装PLANを Cursor で作るときは、毎回 `staffing` 非変更を明示する

## 2. 1業種分の md を作る基本手順
### Step 1. 業種の勝ち筋を1文で定義する
- 例: 「不動産なら、物件提案から契約までの迷いを減らす管理アプリ」
- 例: 「建設なら、安全書類と現場配員の判断が速い管理アプリ」

### Step 2. 通常フローの魅力を決める
- ダッシュで何が一目で分かるべきか
- 一覧カードで何を比べられるべきか
- 詳細ページで何を判断できるべきか
- 次アクションが何に向かうべきか

### Step 3. 技術デモの意味を決める
- OCR / PDF / ナレッジAI / 翻訳 / マッチングAI を「何の業務課題に効くか」で整理する
- 技術を単体で褒めるのではなく、通常フローのどこを補強するかで書く

### Step 4. モバイル / PC の優先順位を書く
- モバイル: 1画面1判断、スクロールしても迷わない構成
- PC: 比較しやすさ、一覧性、複数情報の同時把握

### Step 5. 3者レビューに通る言葉へ整える
- 業種担当者にとって自然か
- Web担当者にとって説明しやすいか
- UI/UX デザイナーにとって情報設計が明快か

## 3. レビュー観点
### 3-1. 業種担当者レビュー
確認したいこと
- その業種特有の業務順序が崩れていないか
- 用語が現場で実際に使われる言い回しに近いか
- 一覧カードや詳細ページの主情報が、判断材料として妥当か
- AI の理由説明が、現場の評価軸や審査軸とつながっているか

合格の目安
- 「これなら自分の顧客にそのまま見せられる」
- 「この順番なら業務の流れとして納得できる」

### 3-2. Web担当者レビュー
確認したいこと
- タイトルと見出しが、商談やLPの見出しに転用しやすいか
- トップで見せる価値と下層ページの内容が矛盾していないか
- 技術デモが“機能紹介”ではなく“導入メリット”として語れるか
- 次アクション導線が、デモのストーリーとしてスムーズか

合格の目安
- 「なぜこの画面が必要か」を1スライドで説明できる
- 「AI を入れる意味」が業務課題に結びついている

### 3-3. UI/UX デザイナーレビュー
確認したいこと
- 各ページの第一情報、第二情報、補助情報の順が明確か
- 一覧→詳細→次アクションの流れが自然か
- モバイルで要素が詰まりすぎないか、PC で比較性があるか
- 通常画面と技術デモ画面の役割が混ざっていないか

合格の目安
- 画面ごとの主目的が1つに絞れている
- 同じ情報が別ページで重複しすぎていない

## 4. 記述の原則
- 先に「誰に何が伝わるか」を書き、その後にページ構成を書く
- UI コンポーネント名ではなく、まず体験・判断・訴求の言葉で書く
- 文言案は“汎用的に正しい”より“その業種で魅力的”を優先する
- ただし誇張しすぎず、既存コードで持てるデータ構造に寄せる

## 5. ページ別に最低限書くこと
| ページ種別 | 最低限書く内容 |
|------------|----------------|
| ダッシュ | 第一印象、主要KPI、上位カード、モバイル/PC差 |
| 一覧 | カードタイトル、比較軸、クリック動機、次アクション |
| 詳細 | 何を判断できるか、どの情報を上に置くか |
| 書類 | 不備、期限、進捗、OCR/PDF との接続 |
| マッチング | スコアの意味、理由の見せ方、要件との整合 |
| ナレッジ | FAQ 軸、シード文、AI の価値説明 |
| 実務 | 1日の動き、担当者目線の安心感、次工程への導線 |

## 6. 技術デモを扱うときの原則
- 技術デモは「本編の代替」ではなく「本編を強化する補助線」として書く
- 1業種につき、主役にする技術デモは多くても2〜3個に絞る
- それ以外の技術デモは「横断価値」「将来拡張」として位置付ける
- `staffing + client` で非表示になるデモがあることを前提に、ロール別の見せ方も書く

## 7. 作成順の推奨
### 第1グループ
- `real-estate`
- `construction`
- `medical`

理由
- 一覧、書類、実務、マッチングの違いが見せやすく、横展開の成功パターンを作りやすい

### 第2グループ
- `professional`
- `sales`
- `logistics`
- `education`

理由
- FAQ、提案、運用、進行管理の文脈差が大きく、テンプレート適用の精度確認に向いている

## 8. 完了条件
各業種 md が次を満たしたら初版完了とする。

- 体験コンセプトが1文で説明できる
- ダッシュ / 一覧 / 詳細 / 書類 / 実務 / マッチング / ナレッジの方針が埋まっている
- 次アクションと技術デモ活用の方針が書かれている
- 業種担当者 / Web担当 / UI/UX デザイナーの確認観点が明記されている

## 9. Cursor 実装PLANへ渡す条件
各業種 md は、次の状態になってから Cursor に実装PLAN作成を依頼する。

- 変更対象ページが列挙されている
- 変更対象ファイルの当たりが付いている
- 変更しないもの、特に `staffing` が明記されている
- モバイル / PC の優先表示が最低限定義されている
- 技術デモの位置づけが、通常フローとの関係で説明できる

この条件が満たされていれば、「md を読み込ませて実装PLANを作り、そのまま実装に進む」ための材料として使える。
