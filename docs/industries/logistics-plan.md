# 物流業種 横展開計画（初版）

## 体験コンセプト
- 一言で言うと: 配車、シフト、入構、遅延対応を止めずに回す物流オペレーション管理デモ
- 管理アプリとしての価値: ドライバー/スタッフ、案件、配車、入構書類、遅延対応を一気通貫で見せる
- AI/技術の価値: 配車候補の理由、遅延時の案内、積載ルール FAQ、多言語連絡支援を見せる

## まず検討するページ
- ダッシュ: 本日出荷数、遅延注意、未配車枠、入構書類未完了を見せる
- 候補者一覧: 対応車種、資格、シフト可否、稼働実績をカードで比較する
- 案件一覧: 配送先、時間帯、積載条件、優先度を示す
- マッチング: 配車候補の理由を、資格・時間帯・経験で説明する
- 書類: 入構申請、免許証、誓約書、配送関連書類の進捗を見せる
- 実務: 配車調整、遅延連絡、再手配の流れを見せる
- ナレッジ: 積載ルール、温度帯、遅延連絡テンプレ、事故対応 FAQ を主軸にする

## タイトル・カード方針
- `labels.client` は配送案件 / 便案件 のどちらが実態に近いか要確認
- 候補者カードでは「対応車種」「資格」「時間帯」を主役にする
- 案件カードでは「配送先」「締切時刻」「条件」を主役にする

## 下層ページの中身方針
- 候補者詳細: 対応可能業務、資格、シフト傾向、注意事項を上位表示
- 案件詳細: 時間指定、積載制約、配送ルートメモ、遅延影響を整理
- 書類ページ: 入構前に不足があると止まる書類を最優先表示にする
- ナレッジページ: 温度管理、危険物、遅延連絡、配送先ルールを中心にする

## 次アクション導線
- ダッシュ → 候補者一覧 / 案件一覧 → マッチング → 実務 / 書類
- 実務 → 遅延連絡 or 再手配
- モバイルでは当日運行判断に必要な情報を最短で見せる

## 技術デモの活用
- マッチングAI: 配車候補の理由説明に最適
- ナレッジAI: 積載・温度帯・事故時対応 FAQ に活用
- 翻訳: 多言語スタッフ連絡に相性が良い
- PDF / OCR: 入構関連資料、誓約書、配送関連書類の補助

## レビュー観点
- 業種担当者: 当日オペレーションの止まりどころがきちんと見えるか
- Web担当者: 「再配車の速さ」「遅延時対応の安心感」を訴求できるか
- UI/UX: モバイル前提でも朝の判断に耐える情報設計か

## `staffing` への影響を出さない条件
- この計画は `logistics` 向けの配車・入構・遅延対応の見せ方に限定する
- 完成済みの派遣コックピットは変更しない
- 共通ファイルを修正する場合も、`logistics` のみへ効く条件分岐または既定値維持で閉じる
